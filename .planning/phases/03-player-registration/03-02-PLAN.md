**Step A — `app.d.ts`:** Add `player: import('$lib/server/schemas/event-schemas.js').Player | null` to the `Locals` interface alongside the existing `user` and `session` fields.

**Step B — `hooks.server.ts`:** After `auth.api.getSession()` populates `locals.user`, add player profile lookup and auto-creation. The logic:

1. If `!locals.user` → set `locals.player = null` and continue.
2. Query `SELECT id, user_id, first_name, last_name, birth_date::text, licence_no FROM player WHERE user_id = $userId LIMIT 1`.
3. If found → parse with `PlayerSchema`, assign to `locals.player`.
4. If NOT found → INSERT a new player row using the user's `name` field (split on first space: parts[0] = first_name, parts.slice(1).join(' ') || parts[0] = last_name). Use placeholder birth_date `'1900-01-01'`. Use `ON CONFLICT DO NOTHING`. Then re-SELECT to get the created row, assign to `locals.player`.
5. All SQL uses `sql<Record<string, unknown>[]>` (project requirement).
6. Validate results with `z.array(PlayerSchema).parse()`.

Important constraints:
- `ON CONFLICT DO NOTHING` is mandatory (race condition guard — multiple concurrent requests on first login).
- No FK on user_id (TEXT field, no `REFERENCES`).
- Name split is best-effort: `const parts = (locals.user.name ?? '').split(' '); const firstName = parts[0] ?? ''; const lastName = parts.slice(1).join(' ') || firstName`.
- Import `PlayerSchema` from `$lib/server/schemas/event-schemas.js`.
- Import `sql` from `$lib/server/db.js`.

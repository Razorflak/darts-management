# Phase 3.2: Player Profile — Context

**Gathered:** 2026-03-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Un utilisateur connecté sans profil Player ne peut pas s'inscrire à un tournoi. Cette phase implémente :
1. Le state global Player dans le layout `(app)` (chargé une fois, disponible partout)
2. La page `/profile/create` — création du profil joueur
3. La page `/profile` — visualisation du profil
4. La page `/profile/edit` — modification du profil (si pas de n° licence)
5. Le banner sur la page événement quand le profil manque
6. La tuile dashboard quand le profil manque

La gestion des inscriptions elle-même reste dans Phase 3 / 03.1.

</domain>

<decisions>
## Implementation Decisions

### State global Player
- Chargé dans `(app)/+layout.server.ts` — disponible sur toutes les pages de l'app
- Le layout expose le profil Player (ou `null`) dans `data` — accessible sans requête supplémentaire côté client
- Quand l'utilisateur clique "S'inscrire" : on lit le state, si `null` → on affiche le banner

### Page événement — banner absence de profil
- Banner rouge visible uniquement si : utilisateur **connecté** ET **pas de profil Player**
- Boutons d'inscription grisés (disabled) quand le banner est visible
- Le banner contient un lien vers `/profile/create`
- Utilisateurs non connectés : comportement inchangé (bouton "Se connecter")

### Page `/profile/create`
- Layout `(app)` standard (navbar normale)
- Titre : **"Compléter mon profil"**
- Composant dédié : **`PlayerCreationForm.svelte`** (nouveau composant — pas MinimumPlayerCreationForm)
- Champs :
  - Prénom *(obligatoire)*
  - Nom *(obligatoire)*
  - Département — `DepartmentSelect.svelte` *(obligatoire)*
  - Date de naissance *(optionnel)*
  - Numéro de licence FFD *(optionnel)*
- Si l'utilisateur a déjà un profil et visite `/profile/create` → redirect vers `/profile`
- Après création réussie : redirect vers `?redirectTo=` (retour au tournoi d'où il venait) ou accueil si absent

### Page `/profile` — visualisation
- Accessible via le menu utilisateur de la navbar ("Mon profil")
- Affiche : nom, prénom, département, date de naissance, n° licence
- Bouton **"Modifier"** visible uniquement si l'utilisateur **n'a PAS** de numéro de licence
- Si numéro de licence présent → tous les champs affichés en lecture seule, pas de bouton Modifier
- Redirect vers `/login` si utilisateur non connecté

### Page `/profile/edit`
- Accessible uniquement depuis le bouton Modifier sur `/profile` (pas de lien direct navbar)
- Mêmes champs que `/profile/create` sauf numéro de licence (non modifiable après saisie ?)
- Si l'utilisateur a un numéro de licence → redirect vers `/profile` (protection serveur)
- Tous les champs modifiables si pas de licence : prénom, nom, département, date naissance

### Tuile dashboard (absence de profil)
- Composant tuile affiché sur le dashboard si l'utilisateur est connecté sans profil Player
- Message : "Pour participer aux évènements de la FFD, créez vous un profil de joueur"
- Bouton : lien vers `/profile/create`
- La tuile disparaît dès que le profil est créé (state global mis à jour)

### Claude's Discretion
- Design exact de la tuile dashboard (style, icône, couleurs)
- Comportement exact du banner (position, dismissible ou non)
- Gestion des erreurs de validation côté serveur

</decisions>

<specifics>
## Specific Ideas

- Le composant `PlayerCreationForm.svelte` est distinct de `MinimumPlayerCreationForm.svelte` — il est plus complet (date naissance + licence en optionnel)
- La règle "licence = champs désactivés" protège l'intégrité des données FFD — à appliquer côté serveur également (pas juste UI)
- Le state global dans le layout évite les requêtes répétées à chaque page

</specifics>

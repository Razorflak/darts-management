## Summary

Phase 03.2 requires three coordinated changes to deliver the player profile gate. First, `hooks.server.ts` must stop auto-creating a player profile on login — currently it creates a minimal profile from the user's display name for every authenticated user on first request. This auto-create behavior conflicts with the requirement that users without a complete profile are blocked and redirected to `/profile/create`. The auto-create must be removed and replaced with a "look up only, never create" approach.

Second, the `(app)/+layout.server.ts` must query and expose `locals.player` in its return value so that every child page can read `data.player` without additional queries. The player profile is already stored in `locals.player` by the (refactored) hooks — the layout simply passes it downstream. This enables the dashboard tile and event page banner to react to the presence/absence of a profile without each page making its own DB call.

Third, four new route files implement the profile UX: `/profile/create` (form + action), `/profile` (read-only view), `/profile/edit` (form + action with licence guard), and the `PlayerCreationForm.svelte` component that is shared between create and edit. The profile creation and update endpoints use SvelteKit form actions (progressive enhancement, no separate API endpoint) following the pattern already used in `(auth)` routes.

**Primary recommendation:** Remove auto-create from hooks.server.ts in the first task of Wave 1, then build layout state propagation and profile pages on top of that clean foundation.

### Pattern 1: hooks.server.ts Refactor — Remove Auto-Create

**What:** Remove the `INSERT INTO player` block from `authHandle`. After refactor, `locals.player` is `null` when no profile exists (instead of auto-creating one).

**When to use:** This is the prerequisite for all other tasks in this phase.

**Current code (to be removed):**
```typescript
// hooks.server.ts — REMOVE this block
```

**After refactor:**
```typescript
// hooks.server.ts — KEEP only the SELECT, set null if no row
const rows = await sql<Record<string, unknown>[]>`
  SELECT id, user_id, first_name, last_name, birth_date::text, licence_no, department
```

**Critical side effect:** Any existing user in the database who has an auto-created player row (with empty/placeholder first/last name) will still have a `locals.player`. Those rows remain valid — no data migration needed. Only new users going forward will get `null`.

### Pattern 2: Layout Server — Expose Player State

**What:** `(app)/+layout.server.ts` adds `player` to its return value. Because hooks already populates `locals.player`, this is a single line addition.

```typescript
// (app)/+layout.server.ts
export const load: LayoutServerLoad = async ({ locals }) => {
  if (!locals.user) {
    redirect(302, "/login")
  }
  const roles = await getUserRoles(locals.user.id)
  const hasAdminAccess = roles.some((r) =>
    ["adminFederal", "adminLigue", "adminComite", "adminClub"].includes(r.role),
  )
  return {
    user: locals.user,
    session: locals.session,
    hasAdminAccess,
    player: locals.player,  // ADD THIS
  }
}
```

**Child pages receive `data.player`** automatically via SvelteKit's layout data propagation — no additional load functions needed in child routes.

### Pattern 3: Profile Creation — SvelteKit Form Actions

**What:** `/profile/create/+page.server.ts` uses a SvelteKit `actions` export (not an API endpoint) to handle the POST.

**Why form actions:** The `(auth)` routes (register, login, reset-password) all use this pattern. It gives progressive enhancement, proper error handling via `fail()`, and redirect via `redirect()`.

```typescript
// profile/create/+page.server.ts
export const load: PageServerLoad = async ({ locals, url }) => {
  if (!locals.user) redirect(302, "/login")
  if (locals.player) redirect(302, "/profile")  // already has profile
  const redirectTo = url.searchParams.get("redirectTo") ?? "/"
  return { redirectTo }
}

export const actions: Actions = {
  default: async ({ request, locals, url }) => {
    if (!locals.user) redirect(302, "/login")

    const data = await request.formData()
    const raw = {
      first_name: String(data.get("first_name") ?? "").trim(),
      last_name: String(data.get("last_name") ?? "").trim(),
      department: String(data.get("department") ?? "").trim(),
      birth_date: String(data.get("birth_date") ?? "").trim() || null,
      licence_no: String(data.get("licence_no") ?? "").trim() || null,
    }

    const result = CreateProfileSchema.safeParse(raw)
    if (!result.success) {
      return fail(400, { error: result.error.issues[0].message, values: raw })
    }

    // INSERT INTO player ... WHERE user_id = locals.user.id ON CONFLICT DO NOTHING
    // Then redirect to redirectTo param
    const redirectTo = url.searchParams.get("redirectTo") ?? "/"
    redirect(303, redirectTo)
  }
}
```

### Pattern 4: Licence Lock Guard — Server-Side Protection

**What:** `/profile/edit/+page.server.ts` guards against users with a licence number accessing the edit page.

```typescript
export const load: PageServerLoad = async ({ locals }) => {
  if (!locals.user) redirect(302, "/login")
  if (!locals.player) redirect(302, "/profile/create")
  if (locals.player.licence_no) redirect(302, "/profile")  // licence = read-only
  return { player: locals.player }
}
```

This protection must be in the `load` function AND in the form `action` — matching the project rule "Permission vérifiée dans load ET action — évite bypass par POST direct sans visiter la page".

### Pattern 5: Event Page Banner — Conditional UI

**What:** The event page `+page.svelte` reads `data.player` from the layout and conditionally renders a banner and disables registration buttons.

**How `data.player` reaches the event page:** SvelteKit merges layout data into page data automatically. Since `(app)/+layout.server.ts` returns `player`, it is available as `data.player` in all child `+page.svelte` files without any change to their load functions.

### Anti-Patterns to Avoid

- **Do not check `locals.player` in child page load functions**: The layout already provides it. Adding a duplicate query per page defeats the "state global" purpose.
- **Do not create a new `/api/profile` endpoint**: Profile create/edit uses SvelteKit form actions, not API endpoints. The `api/` convention is for AJAX-only flows (tournament registration, search, etc.).
- **Do not add `licence_no` field to the edit form**: The CONTEXT.md says licence is not modifiable after entry. The edit form must exclude it.
- **Do not forget the server-side licence guard in the edit action**: UI hiding is not enough — the action must also reject POSTs from users with a licence.

## Common Pitfalls

### Pitfall 1: Stale `locals.player` After Profile Creation

**What goes wrong:** After the user creates their profile via `/profile/create`, the `locals.player` in hooks is populated at request-start from the DB. If the action redirects with a 303, the next request will re-run hooks and correctly load the new player row. But if any in-page fetch (AJAX) is made after creation, `data.player` in the client-side page data is still `null` until a full navigation happens.

**Why it happens:** SvelteKit layout data is server-loaded; it doesn't automatically reactive-update after a client-side fetch.

**How to avoid:** The profile create action uses `redirect(303, ...)` which triggers a full page navigation, ensuring hooks re-runs and `locals.player` is freshly populated. Do not use `invalidateAll()` as a primary mechanism — the redirect is simpler and more reliable.

**Warning signs:** Banner still shows after profile creation = redirect is not happening, or `redirectTo` URL is being used before layout data refreshes.

### Pitfall 2: The `redirectTo` Parameter on `/profile/create` Must Be Encoded

**What goes wrong:** If `/events/[id]` URL is passed raw as `?redirectTo=/events/abc-123`, the `abc-123` is URL-safe, but in general event IDs (UUIDs) contain `-` which is fine. However, the `?redirectTo=` value must be `encodeURIComponent`-encoded in the Svelte template to avoid breaking URL parsing.

**How to avoid:**
```svelte
href="/profile/create?redirectTo={encodeURIComponent($page.url.pathname)}"
```

### Pitfall 3: Auto-Created Players In Production

**What goes wrong:** Existing users already have auto-created player rows with potentially minimal data (first_name/last_name split from display name, null department). After this phase, those users will have `locals.player !== null` and will see `/profile` with incomplete data rather than the create-profile flow.

**Why it matters:** Those users already exist in the DB — the hooks refactor doesn't affect them. They skip the onboarding.

**How to handle:** This is acceptable and intentional. Users with auto-created profiles who lack department can use `/profile/edit` to complete their data. The banner and dashboard tile only show when `player === null`, not when profile is incomplete.

**Warning signs:** If the requirement changes to "profile is complete" rather than "profile exists", additional logic is needed.

### Pitfall 4: `(app)/+layout.server.ts` Type Inference

**What goes wrong:** Adding `player` to the layout's return type requires updating the TypeScript inference. SvelteKit generates `$types` for layout data, so child pages calling `data.player` will typecheck automatically once the layout returns it.

**How to avoid:** Explicitly type the return or rely on `LayoutServerLoad` inference — both work. Do not cast `data.player` in child pages.

### Pitfall 5: `birth_date` Field Handling

**What goes wrong:** The `player` table stores `birth_date` as `DATE` (now nullable per migration 015). In the Zod schema, `birth_date` is `z.string().nullable()` because it's cast to text in SQL (`birth_date::text`). The profile create/edit form must submit it as a text string `YYYY-MM-DD` (from `<input type="date">`), and the insert/update must pass it directly or use `::date` cast.

**How to avoid:**
- Form `<input type="date">` returns `YYYY-MM-DD` format natively — no conversion needed
- Empty string from the form must be coerced to `null` before DB insert
- SQL insert: `${birth_date ?? null}` — postgres.js handles null correctly

## Code Examples

### Profile Create Action — Full Pattern

```typescript
// Source: established project pattern from (auth)/register/+page.server.ts
import { fail, redirect } from "@sveltejs/kit"
import { z } from "zod"
import { sql } from "$lib/server/db"
import type { Actions, PageServerLoad } from "./$types"

const CreateProfileSchema = z.object({
  first_name: z.string().min(1, "Prénom obligatoire"),
  last_name: z.string().min(1, "Nom obligatoire"),
  department: z.string().min(1, "Département obligatoire"),
  birth_date: z.string().nullable(),
  licence_no: z.string().nullable(),
})
export type CreateProfileInput = z.infer<typeof CreateProfileSchema>

export const load: PageServerLoad = async ({ locals, url }) => {
  if (!locals.user) redirect(302, "/login")
  if (locals.player) redirect(302, "/profile")
  return { redirectTo: url.searchParams.get("redirectTo") ?? "/" }
}

export const actions: Actions = {
  default: async ({ request, locals, url }) => {
    if (!locals.user) redirect(302, "/login")

    const data = await request.formData()
    const raw = {
      first_name: String(data.get("first_name") ?? "").trim(),
      last_name: String(data.get("last_name") ?? "").trim(),
      department: String(data.get("department") ?? "").trim(),
      birth_date: String(data.get("birth_date") ?? "").trim() || null,
      licence_no: String(data.get("licence_no") ?? "").trim() || null,
    }

    const result = CreateProfileSchema.safeParse(raw)
    if (!result.success) {
      return fail(400, { error: result.error.issues[0].message, values: raw })
    }
    const { first_name, last_name, department, birth_date, licence_no } = result.data

    await sql`
      INSERT INTO player (user_id, first_name, last_name, department, birth_date, licence_no)
      VALUES (${locals.user.id}, ${first_name}, ${last_name}, ${department},
              ${birth_date ?? null}, ${licence_no ?? null})
      ON CONFLICT DO NOTHING
    `

    const redirectTo = url.searchParams.get("redirectTo") ?? "/"
    redirect(303, redirectTo)
  }
}
```

## Open Questions

1. **Existing auto-created player rows with incomplete data**
   - What we know: Existing dev/staging users already have player rows created by the current auto-create behavior (minimal first/last name, no department)
   - What's unclear: Whether those users should be forced through the profile edit flow on first login after this phase ships
   - Recommendation: Do not force — the phase requirement says "utilisateur sans profil". Users with auto-created profiles already have a row; they can optionally edit. A future enhancement could add a "profile completeness" check if needed.

2. **`redirectTo` security — open redirect risk**
   - What we know: The `redirectTo` parameter is read from URL query string and used in `redirect(303, redirectTo)`
   - What's unclear: Whether validation is needed to prevent open-redirect to external domains
   - Recommendation: Sanitize `redirectTo` to only allow relative URLs (must start with `/`). Simple check: `redirectTo.startsWith("/") ? redirectTo : "/"`. This matches FFD's single-domain use case.

3. **`department` field in `PlayerSchema` — nullable vs required**
   - What we know: `PlayerSchema.department` is `z.string().nullable()`. In the player table, `department` was added as `VARCHAR(50)` nullable. But the profile create form requires it as obligatoire.
   - What's unclear: Whether to add a DB-level NOT NULL constraint or handle only at application level
   - Recommendation: Handle at application level (Zod required in create form). No new migration needed — the department constraint is a product rule, not a data model invariant (admin-created players may have no department).

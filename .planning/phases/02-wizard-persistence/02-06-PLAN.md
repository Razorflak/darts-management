Root cause (from 02-UAT.md):
  EventStep.svelte lines 16-36 — $state initializers for startDateObj, endDateObj, registrationDateObj
  execute ONCE at component mount. When applyTemplate() in +page.svelte does `event = newEvent`,
  the event binding updates, but the local $state Date objects stay undefined (their initializers
  already ran). The $effect blocks then RUN (because event.startDate changed) but they write
  INTO event.startDate FROM the local objects (still undefined) — overwriting the just-applied dates.

Current problematic code in EventStep.svelte (lines 16-36):
  let startDateObj = $state<Date | undefined>(
    event.startDate ? new Date(event.startDate + 'T00:00') : undefined,
  )
  // ... similar for endDateObj and registrationDateObj

  $effect(() => {
    event.startDate = startDateObj ? startDateObj.toISOString().slice(0, 10) : ''
  })
  // ... similar effects writing TO event

Fix strategy: SPLIT each date field into two responsibilities:
  1. A $effect that READS from event.xxxDate and syncs INTO the local Date object
     (direction: prop → local)
  2. A separate handler (on Datepicker change event) that writes FROM local Date object back INTO event
     (direction: local → prop)

This avoids the circular dependency (current $effects write to event, which triggers themselves).

Svelte 5 pattern for reactive prop→local sync:
  $effect(() => {
    startDateObj = event.startDate ? new Date(event.startDate + 'T00:00') : undefined
  })

But this creates a loop: $effect reads event.startDate → sets startDateObj → Datepicker calls
onChange → we write event.startDate → $effect re-runs.

Better pattern — use $effect only for INBOUND sync (prop → local), and use Datepicker's
onchange/onselect callback for OUTBOUND sync (local → prop):

  let startDateObj = $state<Date | undefined>(
    event.startDate ? new Date(event.startDate + 'T00:00') : undefined,
  )

  // Inbound: sync from prop when event is replaced externally (template apply)
  $effect(() => {
    const incoming = event.startDate
    // Only update local if the prop disagrees with what the local Date would produce
    const localIso = startDateObj ? startDateObj.toISOString().slice(0, 10) : ''
    if (incoming !== localIso) {
      startDateObj = incoming ? new Date(incoming + 'T00:00') : undefined
    }
  })

  // Outbound: write back when user picks a date in the Datepicker
  // Use Datepicker's bind:value — but since startDateObj is $state, any change to it
  // from the Datepicker interaction will trigger the $effect again. The guard
  // `incoming !== localIso` prevents the loop.

This is the correct Svelte 5 pattern. The guard (incoming !== localIso) breaks the cycle:
  - Template applies → event.startDate = '2026-05-01' → effect runs → incoming='2026-05-01', localIso='' → sets startDateObj → Datepicker re-renders with new date
  - User picks date → startDateObj = new Date('...') → effect runs → incoming still same → localIso now matches incoming → no-op

Apply the same pattern to endDateObj and registrationDateObj.

Note: The existing Datepicker uses bind:value={startDateObj} which BOTH reads and writes startDateObj.
The $effect inbound sync + bind:value outbound is the correct pattern here.
The outbound write to event.xxxDate should happen via an $effect watching startDateObj:
  $effect(() => {
    event.startDate = startDateObj ? startDateObj.toISOString().slice(0, 10) : ''
  })
But this creates the loop again. The cleanest solution is a single bidirectional $effect with a guard:

  $effect(() => {
    const iso = startDateObj ? startDateObj.toISOString().slice(0, 10) : ''
    if (event.startDate !== iso) event.startDate = iso
  })
  $effect(() => {
    const target = event.startDate ? new Date(event.startDate + 'T00:00') : undefined
    const current = startDateObj ? startDateObj.toISOString().slice(0, 10) : ''
    if (event.startDate !== current) startDateObj = target
  })

This two-$effect approach causes a micro-loop. The SIMPLEST correct approach:

Replace the three $state+$effect pairs with $derived for the Date objects used ONLY for reading,
and use a separate onchange on Datepicker for writing back. But Flowbite Datepicker requires
bind:value (two-way).

RECOMMENDED APPROACH (battle-tested with Svelte 5 + Flowbite):
Keep bind:value on Datepicker. Use a SINGLE $effect per date field that syncs the inbound direction
with a string-comparison guard, and remove the outbound $effects — let the bind:value handle outbound:

Wait — bind:value on a $state variable IS the outbound mechanism. The problem is that the initial
$state value doesn't update. So we need to make startDateObj react to event.startDate changes.

FINAL PATTERN (used in working Svelte 5 codebases):

Remove the $state initializer and the separate $effect. Replace with:
  - Keep $state with initial value (for first render)
  - Add a SINGLE $effect that syncs prop→local using a guard string comparison:

  $effect(() => {
    const propIso = event.startDate || ''
    const localIso = startDateObj?.toISOString().slice(0, 10) ?? ''
    if (propIso !== localIso) {
      startDateObj = propIso ? new Date(propIso + 'T00:00') : undefined
    }
  })

  // Keep outbound: local → prop (runs when user picks a date via Datepicker bind:value)
  $effect(() => {
    event.startDate = startDateObj?.toISOString().slice(0, 10) ?? ''
  })

The guard in the inbound $effect (propIso !== localIso) is what breaks the potential loop:
once both effects have settled, propIso === localIso → inbound $effect is a no-op.

// Local Date objects for Datepicker (two-way via bind:value)
let startDateObj = $state<Date | undefined>(
  event.startDate ? new Date(event.startDate + 'T00:00') : undefined,
)
let endDateObj = $state<Date | undefined>(
  event.endDate ? new Date(event.endDate + 'T00:00') : undefined,
)
let registrationDateObj = $state<Date | undefined>(
  event.registrationOpensAt ? new Date(event.registrationOpensAt + 'T00:00') : undefined,
)

// Inbound sync: event prop → local Date (handles external reassignment e.g. applyTemplate)
$effect(() => {
  const propIso = event.startDate || ''
  const localIso = startDateObj?.toISOString().slice(0, 10) ?? ''
  if (propIso !== localIso) {
    startDateObj = propIso ? new Date(propIso + 'T00:00') : undefined
  }
})
$effect(() => {
  const propIso = event.endDate || ''
  const localIso = endDateObj?.toISOString().slice(0, 10) ?? ''
  if (propIso !== localIso) {
    endDateObj = propIso ? new Date(propIso + 'T00:00') : undefined
  }
})
$effect(() => {
  const propIso = event.registrationOpensAt || ''
  const localIso = registrationDateObj?.toISOString().slice(0, 10) ?? ''
  if (propIso !== localIso) {
    registrationDateObj = propIso ? new Date(propIso + 'T00:00') : undefined
  }
})

// Outbound sync: local Date → event prop (handles user picking a date in the Datepicker)
$effect(() => {
  event.startDate = startDateObj?.toISOString().slice(0, 10) ?? ''
})
$effect(() => {
  event.endDate = endDateObj?.toISOString().slice(0, 10) ?? ''
})
$effect(() => {
  event.registrationOpensAt = registrationDateObj?.toISOString().slice(0, 10) ?? undefined
})

The inbound and outbound guards work together: after template applies, inbound syncs prop→local,
outbound then writes local→prop (same value, no change), inbound sees no difference → stable.

# Phase 03.1 Plan 01: Teams and Doubles Registration — DB Migration and Zod Schemas Summary

**SQL migration introducing team/team_member tables, replacing tournament_registration.player_id with team_id, and updating all Zod schemas and endpoint SQL for the team-based registration model.**

- Migration 013 created: `team` (id, created_at), `team_member` (composite PK, FK RESTRICT on player, CASCADE on team), `player.department VARCHAR(50)`, `tournament_registration.player_id` replaced by `team_id NOT NULL` with UNIQUE(tournament_id, team_id)
- Existing registration data migrated via pure CTE: each pre-existing inscription became a solo team
- All Zod schemas updated: TeamSchema, TeamMemberSchema, PartnerSearchResultSchema added; PlayerSchema, RosterEntrySchema, TournamentRegistrationSchema, PlayerSearchResultSchema updated to team model
- All endpoint SQL and Svelte components adapted to use the new team model — no remaining type errors from schema change

## Decisions Made

- Solo registration always creates a team of 1 — consistent with the "always teams" pattern established in project decisions
- Roster renders all members in a `/`-separated string (e.g., "Dupont Jean / Martin Paul") — appropriate for both solo and doubles display
- Self-unregister finds the team via `NOT EXISTS` (team has no other member) before deleting — prevents accidental team deletion in doubles context

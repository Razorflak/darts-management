Créer le répertoire et le fichier `packages/front/src/routes/tournaments/[id]/partner/search/+server.ts`.

Comportement :
- Accessible uniquement aux joueurs connectés (locals.user + locals.player requis, sinon 401)
- Lit le paramètre `q` depuis url.searchParams
- Si q.length < 3 → retourner json([]) immédiatement (pas de requête DB)
- Requête ILIKE sur first_name, last_name, department avec `'%' + q + '%'`
- Exclure le joueur connecté : WHERE id != ${locals.player.id}
- Ordre : last_name, first_name
- LIMIT 10
- Valider avec z.array(PartnerSearchResultSchema).parse()
- Retourner json(results)

NE PAS vérifier si le tournoi [id] existe — la recherche est globale sur tous les joueurs, pas filtrée par tournoi. Le paramètre [id] est dans l'URL pour cohérence de routing mais n'est pas utilisé dans la query.

```typescript
import { json, error } from '@sveltejs/kit'
import { z } from 'zod'
import { sql } from '$lib/server/db.js'
import { PartnerSearchResultSchema } from '$lib/server/schemas/event-schemas.js'
import type { RequestHandler } from './$types'

export const GET: RequestHandler = async ({ url, locals }) => {
  if (!locals.user || !locals.player) error(401, 'Non authentifié')

  const q = url.searchParams.get('q') ?? ''
  if (q.length < 3) return json([])

  const results = z.array(PartnerSearchResultSchema).parse(
    await sql<Record<string, unknown>[]>`
      SELECT id, first_name, last_name, department
      FROM player
      WHERE id != ${locals.player.id}
        AND (
          first_name ILIKE ${'%' + q + '%'}
          OR last_name  ILIKE ${'%' + q + '%'}
          OR department ILIKE ${'%' + q + '%'}
        )
      ORDER BY last_name, first_name
      LIMIT 10
    `
  )
  return json(results)
}
```

**Fichier 1 : packages/front/src/routes/(app)/events/[id]/+page.server.ts**

Mettre à jour la query des tournois pour remplacer la jointure `r_me.player_id` par une jointure via `team_member` :

```sql
SELECT
  t.id, t.name, t.category, t.check_in_required,
  COUNT(r.id)::int AS registration_count,
  (r_me.id IS NOT NULL) AS is_registered
FROM tournament t
LEFT JOIN tournament_registration r ON r.tournament_id = t.id
LEFT JOIN tournament_registration r_me
  ON r_me.tournament_id = t.id
  AND EXISTS (
    SELECT 1 FROM team_member tm
    WHERE tm.team_id = r_me.team_id AND tm.player_id = ${currentPlayerId}
  )
WHERE t.event_id = ${eventId}
GROUP BY t.id, r_me.id
ORDER BY t.name
```

Note : currentPlayerId peut être NULL (visiteur non connecté sans player) — le EXISTS retournera false naturellement car aucune ligne ne matchera un player_id NULL.

**Fichier 2 : packages/front/src/routes/tournaments/[id]/admin/+page.server.ts**

Remplacer la query roster par une query avec json_agg :

```sql
SELECT
  r.id AS registration_id,
  r.team_id,
  r.checked_in,
  r.registered_at,
  json_agg(
    json_build_object(
      'player_id', p.id,
      'first_name', p.first_name,
      'last_name', p.last_name,
      'department', p.department
    ) ORDER BY p.last_name, p.first_name
  ) AS members
FROM tournament_registration r
JOIN team_member tm ON tm.team_id = r.team_id
JOIN player p ON p.id = tm.player_id
WHERE r.tournament_id = ${params.id}
GROUP BY r.id
ORDER BY r.registered_at
```

La variable `roster` reste `z.array(RosterEntrySchema).parse(rosterRows)` — RosterEntrySchema a déjà z.preprocess pour members (Plan 01).

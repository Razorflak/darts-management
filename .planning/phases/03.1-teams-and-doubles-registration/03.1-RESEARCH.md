## Summary

Phase 3.1 est un refactor breaking de la couche inscription. L'entité centrale passe de `player` à `team` : `tournament_registration.player_id` est remplacé par `team_id`. Ce changement impacte trois surfaces : la migration DB (nouvelles tables + ALTER TABLE + migration des données existantes), les endpoints d'inscription (self-service et admin), et les composants Svelte (modal doubles, mise à jour de PlayerSearch, mise à jour de l'affichage du roster).

Le champ département est ajouté à `player` et à tous les contextes d'affichage de joueur (recherche admin, modal doubles, création de joueur admin). Les catégories doubles déclenchent un flux différent (modal avec recherche de partenaire) ; les catégories simples conservent l'inscription directe mais maintenant via une team solo auto-créée.

La logique de lookup d'une team existante est un point critique : avant de créer une nouvelle team pour deux joueurs, il faut vérifier si une team avec exactement ces deux membres existe en DB. Cette requête est non-triviale et doit s'exécuter dans une transaction pour éviter des doublons de teams sous concurrence.

**Primary recommendation:** Commencer par la migration DB (013_teams.sql) avec migration des données existantes, puis refactorer les endpoints dans l'ordre : self-register → admin-register → partner search endpoint. La modal doubles est le dernier livrable UI.

```sql
-- Step 5: Migrate existing data — create a solo team per existing registration

-- Use a CTE to pair new team ids with existing registrations

  INSERT INTO team (id)
  SELECT gen_random_uuid() AS id
  FROM tournament_registration r
  WHERE r.team_id IS NULL
  RETURNING id

paired AS (
  SELECT
    t.id AS team_id,
    r.id AS reg_id,
    r.player_id
  FROM new_teams t
```

### Pattern 1 : Lookup ou création d'une team existante pour deux joueurs

C'est la requête la plus complexe de cette phase. Avant de créer une team, vérifier si une team {playerA, playerB} existe déjà en DB (dans n'importe quel tournoi, pas seulement le tournoi courant — les teams sont réutilisables).

```typescript
// Source: pattern SQL pur postgres.js
async function findOrCreateTeam(playerIdA: string, playerIdB: string): Promise<string> {
  return await sql.begin(async (tx) => {
    // Look for an existing team that has exactly these 2 members
    const existing = await tx<Record<string, unknown>[]>`
      SELECT tm1.team_id
      FROM team_member tm1
      JOIN team_member tm2 ON tm2.team_id = tm1.team_id AND tm2.player_id = ${playerIdB}
      WHERE tm1.player_id = ${playerIdA}
        AND (SELECT COUNT(*) FROM team_member WHERE team_id = tm1.team_id) = 2
      LIMIT 1
    `
    if (existing.length > 0) {
      return existing[0].team_id as string
    }
    // Create new team
    const [newTeam] = await tx<Record<string, unknown>[]>`
      INSERT INTO team DEFAULT VALUES RETURNING id
    `
    const teamId = newTeam.id as string
    await tx`
      INSERT INTO team_member (team_id, player_id) VALUES
        (${teamId}, ${playerIdA}),
        (${teamId}, ${playerIdB})
    `
    return teamId
  })
}
```

**Important :** La transaction garantit qu'il n'y a pas deux teams identiques créées en concurrence. La contrainte `UNIQUE(team_id, player_id)` sur `team_member` protège aussi contre les membres dupliqués.

### Pattern 3 : Endpoint self-register refactorisé

L'endpoint `/events/[id]/register/+server.ts` reçoit maintenant soit un `team_id` déjà existant (cas doubles — la modal a déjà créé la team via `partner/search`), soit rien (cas indiv — auto-create solo team server-side).

```typescript
// Nouveau schema de requête
const RegisterBodySchema = z.discriminatedUnion('type', [
  z.object({ type: z.literal('solo'), tournament_id: z.string().uuid() }),
  z.object({ type: z.literal('doubles'), tournament_id: z.string().uuid(), team_id: z.string().uuid() })
])
```

**Alternative plus simple :** Ne pas utiliser discriminatedUnion — toujours recevoir un `team_id`. La modal doubles crée la team via un appel préalable (`POST /tournaments/[id]/partner/team`), l'indiv crée la team dans le même POST via `findOrCreateSoloTeam`. Le `team_id` est toujours présent dans le body.

**Recommandation :** Utiliser `team_id` toujours présent. L'API est plus simple et uniforme. La logique de création/lookup se fait dans un utilitaire server-side.

### Pattern 4 : Endpoint de recherche de partenaire

Nouveau endpoint `GET /tournaments/[id]/partner/search?q=` différent de `admin/players/search` car :
- Accessible aux joueurs connectés (pas seulement admins)
- Résultats formatés "Prénom NOM (31 - Haute-Garonne)"
- Filtre sur `department` en plus de `first_name`, `last_name`
- Exclu le joueur connecté (on ne peut pas être son propre partenaire)

```typescript
// /routes/tournaments/[id]/partner/search/+server.ts
const PartnerSearchResultSchema = z.object({
  id: z.uuid(),
  first_name: z.string(),
  last_name: z.string(),
  department: z.string().nullable()
})
type PartnerSearchResult = z.infer<typeof PartnerSearchResultSchema>

export const GET: RequestHandler = async ({ url, locals }) => {
  if (!locals.user || !locals.player) error(401)

  const q = url.searchParams.get('q') ?? ''
  if (q.length < 3) return json([])

  const results = z.array(PartnerSearchResultSchema).parse(
    await sql<Record<string, unknown>[]>`
      SELECT id, first_name, last_name, department
      FROM player
      WHERE id != ${locals.player.id}
        AND (
          first_name ILIKE ${'%' + q + '%'}
          OR last_name  ILIKE ${'%' + q + '%'}
          OR department ILIKE ${'%' + q + '%'}
        )
      ORDER BY last_name, first_name
      LIMIT 10
    `
  )
  return json(results)
}
```

### Pattern 5 : Création d'un nouveau joueur dans la modal doubles

La modal doubles inclut un formulaire inline "Créer un nouveau joueur" (prénom, nom, département). Cela crée un joueur sans `user_id` (même pattern que l'admin), puis crée la team avec ce joueur.

Le POST pour créer un joueur depuis la modal peut aller vers un endpoint dédié ou être géré dans `partner/search` via un POST séparé. Recommandation : endpoint `POST /tournaments/[id]/partner/create` qui crée le joueur et retourne les données formatées, puis la modal fait un appel séparé pour créer la team.

**Alternative plus simple :** Un seul endpoint `POST /events/[id]/register` qui accepte un body avec soit `{ team_id }` (team pré-créée) soit `{ player_id, partner: { first_name, last_name, department } }` pour le cas "nouveau partenaire". Cela réduit le nombre d'endpoints et de round-trips.

### Pattern 7 : Query roster avec json_agg

```sql
SELECT
  r.id AS registration_id,
  r.team_id,
  r.checked_in,
  r.registered_at,
  json_agg(
    json_build_object(
      'player_id', p.id,
      'first_name', p.first_name,
      'last_name', p.last_name,
      'department', p.department
    ) ORDER BY p.last_name, p.first_name
  ) AS members
FROM tournament_registration r
JOIN team_member tm ON tm.team_id = r.team_id
JOIN player p ON p.id = tm.player_id
WHERE r.tournament_id = $1
GROUP BY r.id
ORDER BY r.registered_at
```

Le résultat `members` est un JSONB — appliquer le pattern `z.preprocess` de CLAUDE.md :

### Anti-Patterns to Avoid

- **Inline types :** Toujours dériver de Zod — `z.infer<>` uniquement.
- **Multiples teams identiques :** Sans transaction dans `findOrCreateTeam`, des races conditions créeront des teams dupliquées pour le même binôme. Utiliser `sql.begin()`.
- **`player_id` dans tournament_registration :** Après la migration, plus aucun endpoint ne doit écrire `player_id` dans cette table.
- **FK manquante :** `team_member.player_id` doit avoir `ON DELETE RESTRICT` — ne pas supprimer un joueur qui est dans une team.
- **Recherche partenaire sans seuil :** La recherche commence à 3 caractères (pas 2 comme la recherche admin) — décision produit.
- **`sql<unknown[]>` :** Toujours `sql<Record<string, unknown>[]>` — décision Phase 2-08.

## Don't Hand-Roll

| Problème | Ne pas construire | Utiliser | Pourquoi |
|----------|-------------------|----------|---------|
| Déduplication team doubles | Vérif applicative | Requête SQL dans transaction | Race condition sans transaction |
| Agrégation membres roster | Double requête JS | `json_agg` SQL | Un seul round-trip, pas de N+1 |
| Debounce recherche partenaire | Bibliothèque throttle | `setTimeout`/`clearTimeout` dans `$effect` | Pattern déjà utilisé dans PlayerSearch.svelte |
| Contrainte doublon inscription | Vérif applicative | `UNIQUE(tournament_id, team_id)` + catch 23505 | Atomicité DB |
| Création joueur admin dans modal | Endpoint séparé | Inline dans le POST register | Réduit les round-trips et la complexité |

**Key insight :** La transaction est obligatoire pour `findOrCreateTeam`. Sans elle, deux utilisateurs qui s'inscrivent simultanément avec le même binôme créeront deux teams distinctes pour les mêmes joueurs.

## Common Pitfalls

### Pitfall 1 : Migration des données existantes — mauvaise approche

**What goes wrong :** Utiliser un loop applicatif pour créer une team par inscription existante. Si la migration échoue à mi-chemin, la DB est dans un état incohérent.

**Why it happens :** On pense "je vais itérer les registrations en TS". Mauvaise idée.

**How to avoid :** Tout faire en SQL pur dans la migration :

```sql
-- Créer une team solo pour chaque inscription existante
-- Utilise row_number() pour créer des UUIDs séquentiellement
WITH registrations AS (
  SELECT id AS reg_id, player_id,
         gen_random_uuid() AS team_id
  FROM tournament_registration
),
inserted_teams AS (
  INSERT INTO team (id)
  SELECT team_id FROM registrations
  RETURNING id
),
inserted_members AS (
  INSERT INTO team_member (team_id, player_id)
  SELECT r.team_id, r.player_id FROM registrations r
)
UPDATE tournament_registration tr
SET team_id = r.team_id
FROM registrations r
WHERE tr.id = r.reg_id;
```

**Warning signs :** Migration qui prend plusieurs secondes, ou qui requiert du code TS pour migrer les données.

### Pitfall 2 : Roster query sans GROUP BY correct

**What goes wrong :** `json_agg` sans `GROUP BY r.id` retourne une seule ligne avec tous les membres de toutes les équipes dans un seul agrégat.

**How to avoid :** Toujours `GROUP BY r.id` (et les autres colonnes non-agrégées : `r.team_id`, `r.checked_in`, `r.registered_at`).

**Warning signs :** Le roster retourne 1 ligne au lieu de N.

### Pitfall 3 : team_id dans unregister admin

**What goes wrong :** L'endpoint `admin/unregister` référence encore `player_id` après le refactor. La suppression d'une inscription team doit passer par `team_id`, pas `player_id`.

**How to avoid :** Mettre à jour le schema `UnregisterSchema` pour accepter `team_id: z.uuid()` et la requête DELETE pour cibler `tournament_registration WHERE tournament_id = $1 AND team_id = $2`.

**Warning signs :** Erreur 500 sur DELETE unregister après le refactor.

### Pitfall 4 : Recherche partenaire inclut le joueur courant

**What goes wrong :** Un joueur peut se chercher lui-même comme partenaire, l'inscrire et créer une team avec 2 fois le même joueur (si la contrainte `UNIQUE(team_id, player_id)` n'est pas présente).

**How to avoid :** `WHERE id != ${locals.player.id}` dans la requête de recherche. La contrainte `PRIMARY KEY (team_id, player_id)` sur `team_member` protège en dernier recours.

**Warning signs :** Les résultats de recherche incluent le joueur lui-même.

### Pitfall 5 : `checked_in` sémantique changée

**What goes wrong :** Le check-in était par joueur (`player_id`). Maintenant il est par team. Pour une équipe doubles, checker une team = checker les deux joueurs.

**How to avoid :** `checked_in` reste sur `tournament_registration` (une ligne par team). Pas de check-in individuel par joueur dans cette phase. L'UI affiche "Équipe checkée" ou "Team non présente".

**Warning signs :** Tentative d'ajouter `checked_in` à `team_member`.

### Pitfall 6 : `department` obligatoire mais pas pour les auto-créés

**What goes wrong :** Ajouter `department VARCHAR(50) NOT NULL` fait échouer les hooks.server.ts qui créent des profils auto (ils n'ont pas le département).

**How to avoid :** `department VARCHAR(50)` nullable en DB. Validation Zod `z.string().min(1)` uniquement dans les formulaires de création manuelle. Le schema `PlayerSchema` utilise `z.string().nullable()`.

**Warning signs :** Erreur "null value in column department" lors du premier login d'un utilisateur.

## Code Examples

### Migration SQL complète (013_teams.sql)

```sql
-- Migration 013: teams model, department field, registration refactor

-- 1. Add department to player (nullable — auto-created profiles exempt)
ALTER TABLE player ADD COLUMN department VARCHAR(50);

-- 2. Create team table
CREATE TABLE team (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- 3. Create team_member junction table
CREATE TABLE team_member (
  team_id   UUID NOT NULL REFERENCES team(id) ON DELETE CASCADE,
  player_id UUID NOT NULL REFERENCES player(id) ON DELETE RESTRICT,
  PRIMARY KEY (team_id, player_id)
);
CREATE INDEX team_member_player_idx ON team_member(player_id);

-- 4. Add team_id to registration (nullable initially for migration)
ALTER TABLE tournament_registration
  ADD COLUMN team_id UUID REFERENCES team(id) ON DELETE RESTRICT;

-- 5. Migrate existing data: create one solo team per existing registration
WITH reg_with_teams AS (
  SELECT id AS reg_id,
         player_id,
         gen_random_uuid() AS new_team_id
  FROM tournament_registration
),
_ins_teams AS (
  INSERT INTO team (id)
  SELECT new_team_id FROM reg_with_teams
),
_ins_members AS (
  INSERT INTO team_member (team_id, player_id)
  SELECT new_team_id, player_id FROM reg_with_teams
)
UPDATE tournament_registration r
SET team_id = rt.new_team_id
FROM reg_with_teams rt
WHERE r.id = rt.reg_id;
```

```typescript
// Source: pattern project — sql.begin() avec postgres.js
import { sql } from '$lib/server/db'

export async function findOrCreateSoloTeam(playerId: string): Promise<string> {
  return await sql.begin(async (tx) => {
    const existing = await tx<Record<string, unknown>[]>`
      SELECT tm.team_id
      FROM team_member tm
      WHERE tm.player_id = ${playerId}
        AND (SELECT COUNT(*) FROM team_member WHERE team_id = tm.team_id) = 1
      LIMIT 1
    `
    if (existing.length > 0) return existing[0].team_id as string

    const [team] = await tx<Record<string, unknown>[]>`
      INSERT INTO team DEFAULT VALUES RETURNING id
    `
    const teamId = team.id as string
    await tx`INSERT INTO team_member (team_id, player_id) VALUES (${teamId}, ${playerId})`
    return teamId
  })
}
```

### findOrCreateDoublesTeam (utilitaire server-side)

```typescript
export async function findOrCreateDoublesTeam(playerIdA: string, playerIdB: string): Promise<string> {
  return await sql.begin(async (tx) => {
    const existing = await tx<Record<string, unknown>[]>`
      SELECT tm1.team_id
      FROM team_member tm1
      JOIN team_member tm2
        ON tm2.team_id = tm1.team_id AND tm2.player_id = ${playerIdB}
      WHERE tm1.player_id = ${playerIdA}
        AND (SELECT COUNT(*) FROM team_member WHERE team_id = tm1.team_id) = 2
      LIMIT 1
    `
    if (existing.length > 0) return existing[0].team_id as string

    const [team] = await tx<Record<string, unknown>[]>`
      INSERT INTO team DEFAULT VALUES RETURNING id
    `
    const teamId = team.id as string
    await tx`
      INSERT INTO team_member (team_id, player_id) VALUES
        (${teamId}, ${playerIdA}),
        (${teamId}, ${playerIdB})
    `
    return teamId
  })
}
```

### Endpoint self-register refactorisé

```typescript
// /routes/(app)/events/[id]/register/+server.ts
const RegisterBodySchema = z.object({
  tournament_id: z.string().uuid(),
  // For doubles: partner player_id OR new partner data
  partner_player_id: z.string().uuid().optional(),
  new_partner: z.object({
    first_name: z.string().min(1),
    last_name: z.string().min(1),
    department: z.string().min(1)
  }).optional()
})

export const POST: RequestHandler = async ({ locals, params, request }) => {
  if (!locals.user || !locals.player) error(401, 'Non authentifié')

  const body = RegisterBodySchema.parse(await request.json())
  const selfPlayerId = locals.player.id

  let teamId: string

  if (body.partner_player_id) {
    // Doubles: partner already exists in DB
    teamId = await findOrCreateDoublesTeam(selfPlayerId, body.partner_player_id)
  } else if (body.new_partner) {
    // Doubles: create partner player first
    const [newPlayer] = await sql<Record<string, unknown>[]>`
      INSERT INTO player (first_name, last_name, department, birth_date)
      VALUES (${body.new_partner.first_name}, ${body.new_partner.last_name},
              ${body.new_partner.department}, '1900-01-01')
      RETURNING id
    `
    teamId = await findOrCreateDoublesTeam(selfPlayerId, newPlayer.id as string)
  } else {
    // Solo inscription
    teamId = await findOrCreateSoloTeam(selfPlayerId)
  }

  // Verify tournament is open
  const [row] = await sql<Record<string, unknown>[]>`
    SELECT t.id FROM tournament t
    JOIN event e ON e.id = t.event_id
    WHERE t.id = ${body.tournament_id}
      AND e.id = ${params.id}
      AND e.status = 'ready'
  `
  if (!row) error(404, 'Tournoi introuvable ou fermé')

  try {
    await sql`
      INSERT INTO tournament_registration (tournament_id, team_id)
      VALUES (${body.tournament_id}, ${teamId})
    `
  } catch (err) {
    const pgErr = err as { code?: string }
    if (pgErr.code === '23505') error(409, 'Déjà inscrit')
    throw err
  }

  return json({ ok: true })
}
```
